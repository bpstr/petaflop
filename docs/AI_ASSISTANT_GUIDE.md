# AI Assistant Guide

Petaflop.hu is a Hungarian AI news blog built with Astro 6 from the Aonote theme. Use this guide together with the root `CLAUDE.md` or `AGENTS.md` entry point.

## Project shape

- Framework: Astro 6 with content collections.
- Package manager: npm, with `package-lock.json` committed.
- Runtime: Node.js `>=22.12.0`.
- Production site: `https://petaflop.hu`.
- Deployment output: `dist/`, generated by `npm run build`.

Key files and folders:

- `src/content/posts/` contains blog articles as Markdown or MDX.
- `src/content/tools/` contains AI tool directory entries.
- `src/content/pages/` contains static content pages such as About and 404.
- `src/content.config.ts` defines frontmatter schemas for posts, pages, and tools.
- `src/site.config.ts` contains site title, URL, language, copyright, and homepage limits.
- `src/pages/` contains Astro routes for the homepage, posts, tags, archive, feeds, robots, and tools.
- `src/integrations/aonote-markdown.ts` and `src/plugins/` customize the Markdown pipeline.
- `src/styles/style.css` contains the global theme styles.
- `dist/` and `node_modules/` are generated/dependency directories and should not be hand-edited.

## Commands

Use these commands from the repository root:

```bash
npm install
npm run dev
npm run check
npm run build
npm run preview
```

Command meanings:

- `npm run dev`: start the Astro development server.
- `npm run check`: run `astro check` for TypeScript, Astro, and content validation.
- `npm run build`: remove `.astro` and build the static site into `dist/`.
- `npm run preview`: preview the production build locally.

Run `npm run check` after schema, frontmatter, TypeScript, or Astro component changes. Run `npm run build` before completing changes that affect routes, feeds, layout, content visibility, or deployment behavior.

## Content collections

Posts use the `posts` collection. Required frontmatter:

```yaml
---
title: "Cikk címe"
date: 2026-06-27
summary: "Rövid magyar összefoglaló."
tags: ["AI", "OpenAI"]
draft: false
---
```

Supported post fields:

- `title`: required string.
- `date`: required date.
- `summary`: optional string, recommended for cards and feeds.
- `tags`: optional string array, defaults to `[]`.
- `hidden`: optional boolean, defaults to `false`.
- `draft`: optional boolean, defaults to `false`.
- `cover`: optional string for cover image paths.

Published lists use `getBlogPosts()` from `src/utils/posts.ts`, which excludes posts where `draft` or `hidden` is true and sorts newest first.

Tools use the `tools` collection. Required frontmatter:

```yaml
---
name: ChatGPT
url: https://chat.openai.com
description: AI asszisztens szövegíráshoz, programozáshoz és beszélgetéshez.
category: chat
provider: OpenAI
featured: true
---
```

Allowed tool categories are `chat`, `image`, `design`, `music`, `code`, `research`, `productivity`, `video`, `voice`, `writing`, and `other`.

## Editorial rules

- Write reader-facing blog content in Hungarian.
- Keep the tone clear, factual, and concise. Petaflop.hu covers AI news, not hype copy.
- For current AI news, model releases, company announcements, legal/regulatory changes, prices, benchmarks, or timelines, verify facts against current sources before publishing.
- Prefer primary sources first: company announcements, official docs, research papers, court/regulatory filings, release notes, and conference posts.
- Include source links in the article body or a short `Forrás` line at the end when writing news posts.
- Use exact dates for launches, announcements, reports, and deadlines. Avoid relative wording like "today" unless the post date makes the timing unambiguous.
- Keep titles specific and natural in Hungarian.
- Keep summaries one sentence long when possible.
- Use tags consistently. Common tags include `AI`, `OpenAI`, `Google`, `Anthropic`, `Microsoft`, `videó`, `kódolás`, `ügynökök`, and `kutatás`.
- Do not publish drafts accidentally. Use `draft: true` for unfinished posts.
- Keep filenames lowercase, descriptive, and slug-like, for example `openai-sora-leallitas.md`.

## Astro and code conventions

- Use Astro content collections through `astro:content` APIs such as `getCollection`.
- Keep content schema changes in `src/content.config.ts` and update existing content if a schema change requires it.
- Preserve `siteHref()` and `absoluteUrl()` helpers when creating internal and canonical links.
- Keep deployment configuration aligned with `src/site.config.ts`, especially `baseUrl` and `repoSubpath`.
- Keep UI strings and language-related changes near `src/i18n.ts` and `src/site.config.ts`.
- Prefer existing components, layouts, utilities, and Markdown plugins over new abstractions.
- Avoid adding client-side JavaScript unless the feature genuinely needs browser interactivity.
- Keep generated feeds and sitemap behavior in mind when changing post visibility, URLs, or metadata.

## Images and assets

- Store static post images under `public/static/posts/`.
- Use descriptive filenames and keep source attribution clear when required.
- Do not hotlink images from third-party sites unless explicitly acceptable for the use case.
- When adding image references in Markdown frontmatter or content, confirm the path works in a production Astro build.

## Before handing work back

Check the scope of the change and run the relevant validation:

- Content-only post/tool changes: `npm run check`.
- Route, layout, feed, schema, config, or integration changes: `npm run check` and `npm run build`.
- Visual changes: inspect locally with `npm run dev` or `npm run preview` when practical.

Report any command you could not run and why.
